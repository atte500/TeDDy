# Specification: Execution Reports

## 1. Overview

This document specifies the two types of execution reports generated by `teddy`. While both reports are generated from the same underlying data model and Jinja2 template (`src/teddy_executor/core/services/templates/execution_report.md.j2`), they serve distinct purposes for the two primary workflows.

-   The **Session Report** is a rich, historical artifact for the interactive workflow.
-   The **CLI Report** is a concise, ephemeral payload for the manual, clipboard-driven workflow.

---

## 2. Report Types

### 2.1. The Session Report (`report.md`)

-   **Workflow:** Interactive Session (`teddy resume`, `teddy execute <plan_file>`)
-   **Purpose:** To serve as a complete, auditable, and permanent historical record of a turn's execution. This file is written to the turn directory (e.g., `01/report.md`).
-   **Generation:** Produced when the template formatter is called with `is_concise=false`.

#### Key Characteristics:

-   **Comprehensive:** It includes the full `Rationale` and original `Action Plan` from the `plan.md` that was executed, providing complete context for the turn.
-   **State-Aware:** It does **not** embed the content of successful `READ` actions. In the interactive workflow, this content is passed to the next turn via the session's context management system (`T_next/turn.context`), not through the report.

### 2.2. The CLI Report (stdout / clipboard)

-   **Workflow:** Manual / Stateless (`teddy execute` with plan from clipboard)
-   **Purpose:** To serve as a concise, self-contained, and ephemeral payload that can be copied from the CLI and pasted to an LLM to inform the next turn.
-   **Generation:** Produced when the template formatter is called with `is_concise=true`.

#### Key Characteristics:

-   **Concise:** It **omits** verbose, historical sections from the original plan, such as the `Rationale` and the full `Action Plan`, focusing only on the execution results.
-   **Self-Contained:** It **must include** all information necessary for the next turn. This includes:
    -   The full, verbatim content of any successful `READ` actions.
    -   The current, actual content of any files that caused a `CREATE` or `EDIT` action to fail.

---

## 3. Shared Document Structure

Both report types share a common base structure. Sections are rendered conditionally based on the data from the execution.

1.  **Report Header:** High-level summary (Status, Timestamps).
2.  **Validation Errors (Conditional):** Appears only if pre-flight validation fails.
3.  **Resource Contents (Conditional):** Behavior depends on the report type.
4.  **Action Log:** A detailed, action-by-action log of the execution.

### 3.1. EDIT and CREATE Diffs
To ensure transparency during fuzzy matching and file overwrites, the system must include a unified diff in the `Action Log` for any `EDIT` action where the `Similarity Score` is less than 1.0 and any `CREATE` action where `Overwrite` = `true`. This allows the AI to verify the exact changes applied.

### 3.2. Automated Skipping (Soft Isolation)
If a plan contains multiple actions, any "Terminal Action" (`PROMPT`, `INVOKE`, `RETURN`) is automatically skipped by the orchestrator.
- **Status:** `SKIPPED`
- **Reason:** Action must be executed in isolation to ensure state consistency.

The precise rendering logic and format are defined by the canonical Jinja2 template. This specification serves as a high-level guide to its behavior.
