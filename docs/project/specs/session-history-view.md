# Specification: Session History Log (history.log)

## 1. Overview

This specification defines the `history.log` file – a plain, chronological log of all turn activity in a session. The log is a pure capture of the console output already being printed during session execution; no new log lines are added.

## 2. Guiding Principles

- **Console Fidelity:** The log records exactly what the user sees printed to the console (both stdout and stderr) during execution, in the same format and order.
- **No New Output:** The log is produced by capturing existing output streams without adding any new `typer.echo()`, `print()`, or logging statements. The session behavior is completely unchanged.
- **Simplicity & Auditability:** The log is a sequential, append-only record that mirrors the terminal output for easy skimming and searching.
- **Mode Independence:** The same log content is generated in both `-y` and interactive TUI modes. The TUI's alternate screen buffer does not interfere with the captured output.

## 3. Core Requirements

### 3.1. Artifact Definition

- **File Name:** `history.log`
- **Location:** Session root directory (e.g., `.teddy/sessions/<session_name>/history.log`).
- **Format:** Plain text (not Markdown) – exactly as printed to the console.
- **Capture Scope:** Both `sys.stdout` and `sys.stderr` output. The log file contains the interleaved output from both streams in the order it was printed.
- **Generation:** Automatically generated by the `SessionOrchestrator` during turn execution. The log is opened in append mode at the start of `execute()` and closed when the method exits (via `Tee` context manager). The same content is written to the log file as is printed to the terminal.

### 3.2. Exact Format

The log has **no predefined format beyond the exact console output**. Since it is a pure capture of existing output, the content mirrors exactly what appears on the terminal during the turn execution. This includes:

- Any metadata lines printed by the session infrastructure (e.g., session termination messages)
- Action log lines (action type, description, status)
- Error messages
- Any other console output produced during execution

No header, separator, or formatting is injected by the Tee itself.

### 3.3. Generation Mechanism

The `history.log` is produced by installing a `Tee` context manager that replaces both `sys.stdout` and `sys.stderr` with custom writers at the start of `SessionOrchestrator.execute()`. The Tee:

1. Saves references to the original `sys.stdout` and `sys.stderr`.
2. Opens `history.log` in append mode (`"a"`, UTF-8 encoding).
3. Creates two proxy writer instances (stdout and stderr), each of which writes to both the original stream and the log file.
4. Replaces `sys.stdout` and `sys.stderr` with these writers.
5. On completion (or exception), restores the original streams and closes the log file.

The Tee is installed **before any console output-producing code** (plan parsing, context preparation, validation, execution) so that ALL output for the turn is captured.

### 3.4. Edge Cases

- **File open failure:** If `history.log` cannot be opened (permissions, disk full), the Tee logs a debug warning and does not install. The session continues without a history.log.
- **Write failure:** If writing to the log file fails mid-turn, the exception propagates. The session fails, but console output to the terminal is unaffected (the proxy writer writes to the original stream first).
- **Validation failure:** The console output printed during the validation step is captured (e.g., error messages printed to stderr). The Tee is active even if execution does not proceed to action execution.
- **Communication turn (`## Message`):** The Tee captures whatever output is printed for the message turn (typically nothing beyond the turn transition messages).
- **Non-session mode:** The Tee is only installed when `is_session` is True. Standalone `teddy execute` calls do not produce a `history.log`.
- **`--yolo` vs interactive:** The same capture mechanism works in both modes; the TUI's alternate screen buffer does not suppress console output that is printed before the TUI launches or after it exits.
- **Session branching:** If a session directory is copied, the `history.log` is copied with it. Each branch has its own log continuing from the branch point.
- **Unicode characters:** The file is opened with `encoding="utf-8"`.
- **Flush on each write:** Ensures no data loss on crash.
- **Stdout/stderr restoration:** After Tee exits, the original streams are restored. Exception safety ensures this always happens via `try/finally`.

### 3.5. Implementation Strategy

1. **Create Tee utility:** Implement a `Tee` class in `src/teddy_executor/core/utils/io.py` that is a context manager replacing both `sys.stdout` and `sys.stderr` with proxy writers.
2. **Install in SessionOrchestrator:** At the start of `SessionOrchestrator.execute()`, when `is_session` is True, install the Tee with the `history.log` path derived from `plan_path`. Wrap the rest of the method in `try/finally` to ensure cleanup.
3. **No header printing:** The old requirement to print a metadata header is removed. The Tee captures whatever is already printed — no new `typer.echo()` calls are added.
4. **Testing:**
   - Unit tests for the Tee class (write propagation for both streams, flush, isatty, context manager restore, exception safety).
   - Integration tests for `SessionOrchestrator` verifying that `history.log` contains the expected console output, that it works for both stdout and stderr, and that cleanup is correct.

## 4. Comparison with Old Specification

**Old `session-log.md` spec (to be fully replaced):** Used complex graph reconstruction, full `plan.md` content, and action log sections from `report.md`. Not yet implemented.

**New `history.log` spec:** Plain, linear, console-fidelity format. Implemented via tee'ing existing console output. Much simpler and zero additional data processing.

## 5. Guidelines for Implementation

1. **Minimal changes to existing code:** Do not refactor the entire console printing pipeline. Add a small `Tee` helper class or modify the printing functions to also write to the log file.
2. **File path discovery:** The session root is already known from the `SessionManager` service. The log file path should be constructed once and cached.
3. **Testing:** Unit tests should verify that the log file contains the expected lines after executing a plan. Integration tests should cover both `-y` and interactive (TUI) flows by simulating console output.
4. **Performance:** The log file is small per turn; overhead is negligible.
5. **Branching:** If a session is branched by copying the directory, the `history.log` is also copied. This is acceptable; each branch will have its own log continuing from the branch point.

The current session structure, with each turn in a separate directory, is optimized for detailed auditing but makes it difficult to get a high-level, chronological overview of a session's progression. This specification defines a new, automatically generated artifact: the Session Log. This single, consolidated Markdown file presents the entire session history in a clean, readable, "chat-like" format.

## 2. Guiding Principles

-   **Readability First:** The output should be optimized for human consumption, summarizing the key events of the session in chronological order.
-   **Focus on Intent and Outcome:** The view should highlight the AI's plan (`plan.md`) and the factual result of that plan (the `Action Log` from `report.md`), omitting verbose, intermediate artifacts.
-   **Alignment with Core Philosophy:** The feature should align with the project's "Local-First" and "Markdown as Interface" principles, favoring plain-text, portable formats.

## 3. Core Requirements (The "What")

### 3.1. Artifact Definition

1.  **File Name:** `session-log.md`
2.  **Location:** The file will be created at the root of the session directory (e.g., `.teddy/sessions/<session_name>/session-log.md`).
3.  **Generation:** This file is automatically created and appended to by the `teddy execute` command upon the successful completion of each turn's execution. It is a living document that grows with the session.
    -   **Exclusion Rule:** Turns that fail the pre-flight validation step are **not** considered "completed" and will be excluded from the `session-log.md`. Only turns that are successfully validated and executed will be logged.

### 3.2. History Reconstruction Logic

The session log must be resilient to manual user actions (renaming, branching). The generation logic will use a hybrid approach that combines metadata for robustness with folder names for user-friendly display.

1.  **Build Dependency Graph:** The system will scan all turn directories and read their `meta.yaml` files to build a dependency graph using the `turn_id` and `parent_turn_id` fields. This correctly maps out all branches.
2.  **Sort Turns:** Within each branch, turns will be sorted chronologically using the `creation_timestamp` from their `meta.yaml`. *(Note: We rely on this explicit timestamp rather than the OS-level file modification time (`mtime`) because `mtime` is fragile and is not preserved across `git clone` operations. If two sibling branches have an identical `creation_timestamp` due to a user manually copying a turn folder, the system falls back to sorting alphabetically by folder name to break the tie.)*
3.  **Generate Log:** The system will traverse the sorted graph to generate the final `session-log.md`, ensuring all branches are represented in a logical, chronological order.

This ensures the underlying logic is robust and handles branching correctly, while still allowing for flexible sorting and display.

### 3.3. Content Structure

To ensure a clean, readable document, each turn will be appended using the following hierarchical format, separated by a horizontal rule (`---`).

1.  **Turn Header (H2):** A single Level 2 header will be created for the turn. It will use the **turn's folder name** as the primary identifier, followed by the title from the plan's H1 header. This provides a direct visual link to the file system while maintaining a good document structure.
    -   *Example:* `## Turn: 02-branch - Refactor the Core Service`

2.  **Plan Details:** The full content of the `plan.md` file, **excluding** its top-level H1 header, will be included. This preserves the plan's metadata, rationale, and action plan while fitting correctly into the document's structure.

3.  **Action Log (H3):** Extracted verbatim from the corresponding `report.md`.

## 4. Implementation Guide

The generation process will be a pure Python function that takes a session directory path as input and writes the `session-log.md` file as output. The core logic can be broken down into four distinct steps.

### Step 1: Data Ingestion and Graph Construction

1.  **Scan Directories:** Use `pathlib.Path.glob` to find all subdirectories within the session directory.
2.  **Parse Metadata:** For each directory, attempt to read and parse its `meta.yaml` file using the `PyYAML` library. If `meta.yaml` is missing or invalid, the directory is skipped.
3.  **Build Turn Nodes:** Store the parsed metadata, along with the directory path (for later use as the display name), in a simple data structure (e.g., a dictionary or a dataclass). A dictionary mapping `turn_id` to this turn object will be the primary data structure.
4.  **Build Adjacency List:** Create an adjacency list (e.g., a `dict` mapping a `turn_id` to a `list` of its children `turn_id`s) by iterating through all turn nodes and linking each `parent_turn_id` to its child.

### Step 2: Identify Branches and Sort

1.  **Find Roots:** Identify all root nodes of the graph (turns with no parent).
2.  **Find Leaves:** Identify all leaf nodes (turns with no children).
3.  **Trace Branches:** For each leaf node, perform a traversal up to its root using the parent links. This reconstructs the linear history for each individual branch. The turns within each branch will be sorted chronologically using the `creation_timestamp`.

### Step 3: Generate Markdown Content

1.  **Iterate Branches:** Process each reconstructed branch one by one. If there's more than one branch, a top-level header (e.g., `## Branch starting from Turn: [root_folder_name]`) could be added to delineate them.
2.  **Process Turns:** For each turn in a branch:
    -   Read the `plan.md` file.
    -   Extract its H1 title and the rest of its content.
    -   Read the `report.md` file (if it exists).
    -   Extract the content of the `### Action Log` section. This can be done with a simple string search or a more robust Markdown parser if needed, but string splitting should suffice initially.
    -   Format the extracted content into the string template defined in the specification.

### Step 4: Write to File

1.  **Concatenate:** Join all the generated Markdown strings for each turn.
2.  **Write:** Overwrite the `session-log.md` file in the session's root directory with the complete, newly generated content.

### Key Libraries & Dependencies

-   **`pathlib`**: (Standard Library) For robust file system scanning.
-   **`PyYAML`**: For parsing `meta.yaml` files. This will be the only new external dependency required.

### Edge Cases to Handle

-   **Missing Files:** The logic must gracefully handle cases where `meta.yaml`, `plan.md`, or `report.md` are missing for a given turn directory.
-   **Orphaned Turns:** Turns whose `parent_turn_id` does not correspond to any existing turn will be treated as roots of their own branches.
-   **Circular Dependencies:** While unlikely, the graph construction logic should ideally detect and handle circular dependencies to prevent infinite loops (e.g., by keeping track of visited nodes during traversal).
