# Bug: TUI `m` user message duplicates into action log when it should only appear in User Request section

- **Status:** Resolved
- **Milestone:** N/A
- **Vertical Slice:** N/A
- **Specs:** N/A

## Symptoms

When a user provides a reply to a **MESSAGE action** (generated by the LLM during planning), that user reply appears in `report.md` both:
1. In the **Action Log** as the "User Reply" for that MESSAGE action — **this is correct behavior.**
2. As the **"User Request"** (top section header) — **this is the bug.**

The **User Request** section (rendered from `ExecutionReport.user_request`) is intended to show only the optional message provided by the user via the `m` key in the TUI plan reviewer. It should NOT be populated by replies to MESSAGE actions.

The **Action Log** correctly captures the user's typed reply to a MESSAGE action (as `User Reply` in the action's details). However, that same reply should NOT also populate the `user_request` header field.

## Context & Scope

### Regressing Delta
The root cause is in `execution_orchestrator.py`'s `_handle_action_in_loop` method, which unconditionally sets `plan.metadata["user_request"] = captured_message` when `captured_message` is non-empty. For MESSAGE actions, `captured_message` is the user's reply (returned by `confirm_and_dispatch` as `action_log.details`), causing that reply to also appear in the User Request section.

### Environmental Triggers
- Using TUI mode (`teddy start` without `--no-tui`)
- Pressing `m` to input an additional message during plan review (this message should appear in User Request section only).
- LLM generates a MESSAGE action and user provides a reply during execution (this reply should appear in action log only).

### Ruled Out
- The TUI `m` input flows correctly through the `message` parameter of `session_orchestrator.execute()` to `ReportAssemblyData.message`. This path is not the cause of the duplication.
- The Jinja2 template correctly renders MESSAGE action details in the action log and `user_request` in the header. The template is not at fault.
- The ReportAssembler correctly uses `data.message or data.plan.metadata.get("user_request")`. The logic is correct; the bug is that the metadata key is set for MESSAGE actions when it should not be.

## Diagnostic Analysis

### Causal Model
**Data Flow for TUI `m` input:**
1. User presses `m` in TUI → message stored in `app._user_message_cache`
2. On plan submit, the TUI returns the modified plan (not the message)
3. The `message` parameter is passed to `session_orchestrator.execute()` separately → `ReportAssemblyData.message` → `ExecutionReport.user_request`

**Data Flow for MESSAGE action reply:**
1. LLM generates a MESSAGE action during planning
2. During execution, `_dispatch_single_action` skips the TUI reviewer (since `plan.is_communication_turn()` is True)
3. Falls through to `confirm_and_dispatch`, which for MESSAGE actions returns `(action_log, action_log.details)` – the user's typed reply
4. This `captured_message` (the user's reply) is returned to `_handle_action_in_loop`
5. `_handle_action_in_loop` sets `plan.metadata["user_request"] = captured_message` (line ~184)
6. The assembler uses `data.message or data.plan.metadata.get("user_request")` — if `data.message` is None (no `m` input), the MESSAGE reply becomes `user_request`
7. The template renders `user_request` in the User Request section AND the MESSAGE action's `details` in the action log → **duplicate**

**Root Cause:** In `execution_orchestrator.py`, `_handle_action_in_loop` unconditionally sets `plan.metadata["user_request"] = captured_message` for ALL actions, including MESSAGE actions. For MESSAGE actions, `captured_message` is the user's reply to the LLM's message, which should only appear in the action log (as `action_log.details`), not in the User Request section.

### Discrepancies
- [ ] `_handle_action_in_loop` sets `plan.metadata["user_request"]` for MESSAGE actions, causing the user's reply to appear in both User Request and action log. (Resolved: The fix is to skip setting `user_request` for MESSAGE actions.)
- [ ] The TUI `m` input does NOT flow through `_dispatch_single_action`; it flows through the `message` parameter. (Resolved: This was a misunderstanding; the two paths are separate.)

## Solution

### Root Cause
In `execution_orchestrator.py`, `_handle_action_in_loop` unconditionally stores the `captured_message` into `plan.metadata["user_request"]` for ALL actions (line 109: `if captured_message:`). For MESSAGE actions, `captured_message` is the user's typed reply (returned by `confirm_and_dispatch` as `action_log.details`). This causes the same text to appear in both:
- The **User Request** header (via `data.message or data.plan.metadata.get("user_request")` in `execution_report_assembler.py`)
- The **Action Log** user reply for that MESSAGE action

### Fix
Change the condition in `_handle_action_in_loop` to skip MESSAGE actions:

```python
# Before:
if captured_message:
    plan.metadata["user_request"] = captured_message

# After:
if captured_message and action.type.upper() != "MESSAGE":
    plan.metadata["user_request"] = captured_message
```

### Preventative Measures
- MESSAGE action replies are now only stored in `action_log.details` (their correct location).
- The `user_request` field is populated by:
  1. The `message` parameter of `ExecutionOrchestrator.execute()` (from TUI `m` key)
  2. Non-MESSAGE action `captured_message` (from TUI `m` during review)
- A regression test (`test_bug_16_message_user_request.py`) verifies this behavior and prevents future regressions.
