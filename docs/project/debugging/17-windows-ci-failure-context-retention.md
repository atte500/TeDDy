# Bug: Windows CI Failure (Context Retention)
- **Status:** Resolved
- **Milestone:** [10-interactive-session-and-config](../milestones/10-interactive-session-and-config.md)
- **Vertical Slice:** [00-04-context-management-ui](../slices/00-04-context-management-ui.md)

## Symptoms
Expected: Windows CI passes after path normalization.
Actual: Windows CI fails in run 25725368528.

## Context & Scope
### Regressing Delta
Recent changes in `SessionService` and `SessionPruningService` to handle context retention and path normalization for Windows.

### Environmental Triggers
- OS: Windows (CI)

### Ruled Out
- Linux/macOS (assumed passing as those jobs usually precede or run alongside Windows).

## Diagnostic Analysis
### Causal Model
The `SessionOrchestrator` performs pre-flight validation on plans before invoking the `IPlanReviewer`. This validation relies on `is_path_in_context` to verify if files for `READ` or `PRUNE` actions exist in the current session context.

On Windows, paths frequently contain backslashes (`\`). The current normalization logic in `is_path_in_context` (introduced in Case File 16) performs `.lstrip("/").replace("\\", "/")`. This fails in two ways:
1. **Order of Operations:** Stripping occurs *before* the backslash-to-forward-slash replacement. A leading backslash (e.g., `\README.md`) is preserved during the strip and then converted to `/README.md`.
2. **Relative Prefixes:** It fails to handle `.\` or `./` prefixes commonly generated by platform-specific path resolution (e.g., `.\README.md`).

This results in a mismatch against the canonical `README.md` format used in plans, triggering a validation failure.

In Session Mode, a validation failure causes the orchestrator to exit early and trigger a re-plan turn, bypassing the TUI reviewer entirely. This explains why the reviewer call count is 0 in Windows CI logs.

### Discrepancies
- `Target: README.md | Context: ['\\README.md'] -> False`. Conflict: These are semantically identical paths in TeDDy. (Resolved: Normalization order is incorrect; `replace` must precede `lstrip` or both slashes must be stripped).
- `Reviewer.review` called 0 times on Windows. Conflict: Interactive mode is enabled. (Resolved: Early exit due to validation failure).

### Investigation History
1. Initializing case file. Identifying failing CI run (25725368528).
2. Analyzed logs: Identified `test_auto_pruning_heuristics_acceptance` failing on Windows with 0 reviewer calls.
3. Analyzed `is_path_in_context` via `spikes/debug/17-windows-diag.py`. Discovered that leading backslashes are not stripped because `lstrip("/")` is called before `replace("\\", "/")`.
4. Verified fix in `spikes/debug/17-mre-fix-verified.py` by swapping normalization order and handling relative dot-slash prefixes.

## Systemic Audit
- **Vulnerable Pattern:** `path.lstrip("/").replace("\\", "/")` is dangerous because it preserves leading backslashes as forward slashes.
- **Affected Components:**
    - `src/teddy_executor/core/services/validation_rules/helpers.py`: `is_path_in_context` (Primary cause)
    - `src/teddy_executor/core/services/session_service.py`: `_extract_resource_path`
- **[DEBT]:** Found inconsistent use of `lstrip("/")` across `ContextService` and `SessionRepository`. A centralized `normalize_path` utility should be used for all root-relative path canonicalization in a future refactor.

## Solution
The root cause is a platform-specific path normalization mismatch. The fix involves implementing a robust, order-correct normalization logic in `is_path_in_context` and `SessionService`:

```python
def normalize(p):
    return p.replace("\\", "/").lstrip("./").lstrip("/")
```

This ensures all backslashes are forward-slashes before stripping leading characters, and specifically handles the `.\` and `./` relative prefixes.

Verified in `spikes/debug/17-mre-fix-verified.py`.
