## Symptoms

### Symptom 1: "searched None and None" warning
- **Expected:** When starting a session, the system prompt for the agent should be resolved correctly (e.g., `pathfinder.xml`).
- **Actual:** A warning is logged: `PromptManager: Failed to resolve system prompt for agent 'Pathfinder' (searched None and None)`. The agent name uses capital P (`Pathfinder`) while the prompt file is `pathfinder.xml` (lowercase p). This causes `_find_prompt_file` to fail to match the stem.
- **Reproduction:** Run `teddy start` and observe the warning in the console.

### Symptom 2: "Initial Request:" printed twice
- **Expected:** The "Initial Request:" header and its content should appear exactly once during session start.
- **Actual:** The output shows "Initial Request:" twice (first during session creation output, then again after the turn header). The user message content also appears duplicated.
- **Reproduction:** Run `teddy start` with a message and observe the console output.

## Diagnostic Analysis

### Causal Model

#### Symptom 1: "searched None and None" warning (RESOLVED via casefold)
- `PromptManager.fetch_system_prompt()` receives `agent_name = 'Pathfinder'` (capital P).
- `_find_prompt_file()` searches `.teddy/prompts/` for a file whose `Path(f).stem == agent_name`.
- The prompt file is `pathfinder.xml` (stem `pathfinder`, lowercase p).
- The equality check is case-sensitive → fails, returns `None`.
- Both `session_root_prompt` and `teddy_prompt_path` are `None` → warning logs "searched None and None".
- **Fix applied:** Changed `==` to `.casefold()` comparison in `_find_prompt_file`.

#### Symptom 2: Missing "Initial Request:" output (first fix: path correction)
- The surviving call in `SessionLifecycleManager._handle_planning_and_execution()` passes `plan_path = Path(turn_dir).as_posix()` (the turn directory itself, e.g., `.teddy/sessions/xxx/01`).
- `_print_initial_request` resolves the `initial_request.md` file using `Path(plan_path).parent.parent`.
- `.parent` from turn dir → session root (one level up).
- `.parent.parent` from turn dir → sessions root (two levels up) — wrong level.
- The `initial_request.md` file lives at `{session_root}/initial_request.md`, one level above the turn dir.
- So the file is never found → no output.
- **Fix applied:** Changed `parent.parent` to `parent` in `_print_initial_request`.

#### Symptom 3: "Initial Request:" printed on every turn (second fix: first-turn guard)
- After fixing path resolution, `_print_initial_request` is called **unconditionally** in `_handle_planning_and_execution()` for every turn (01, 02, 03...).
- The `initial_request.md` file persists at session root and is found on every turn.
- **Fix applied:** Added guard `if Path(turn_dir).name == "01":` so the call only executes for the very first turn.

#### Note on Duplicate "Initial Request:" in earlier sessions
- When both the lifecycle manager AND the orchestrator called `_print_initial_request`, the output was duplicated (first before turn header, then after). The orchestrator calls were removed in a previous fix, leaving only the lifecycle manager call. The current bug (missing output) emerged because the surviving lifecycle manager call used the wrong path resolution for the file read.

## Solution

### Root Cause 1: Case-Sensitive Prompt File Matching
`PromptManager._find_prompt_file` compares `Path(f).stem == agent_name` (case-sensitive). When the agent name is provided with a capital letter (e.g., "Pathfinder" from user config or CLI) but the prompt file is lowercase ("pathfinder.xml"), the comparison fails, returning `None`.

**Fix:** Changed `==` to `.casefold()` comparison in `_find_prompt_file`:
```python
if Path(f).stem.casefold() == agent_name.casefold():
```

This aligns with `get_available_agents` which already uses `.casefold()` for sorting.

### Root Cause 2: Wrong Path Resolution in `_print_initial_request`
After removing the duplicate `_print_initial_request` calls from the orchestrator (previous fix), the surviving call in `SessionLifecycleManager._handle_planning_and_execution()` passes `plan_path = Path(turn_dir).as_posix()` (the turn directory itself). The function resolves the `initial_request.md` file using:
```python
import_path = Path(plan_path).parent.parent / "initial_request.md"
```
Since `plan_path` is the turn directory (e.g., `.teddy/sessions/xxx/01`), `.parent` gives session root, but `.parent.parent` gives the sessions root — two levels up instead of one. The `initial_request.md` file lives at session root, so the lookup fails silently, producing no output.

**Fix:** Changed `parent.parent` to `parent` in `_print_initial_request`:
```python
import_path = Path(plan_path).parent / "initial_request.md"
```

### Preventative Measures
- **Case normalization policy:** Establish a convention that agent names should be treated case-insensitively across the codebase. `_find_prompt_file` was the only location with a strict equality check; all other comparisons (sorting, display) already used casefold or case-insensitive methods.
- **Code review guard:** Any new comparison between user-provided agent names and filesystem stems must use casefold or lowercasing.
- **Path resolution audit:** When resolving session-level files from a turn directory, verify the number of `.parent` calls matches the directory depth. Prefer using `turn_path.parent` (session root) explicitly rather than chaining `.parent.parent` from a variable that might be a file path or directory.
- **Lifecycle duplication:** When adding new output in session lifecycle, audit whether the same output is already produced by the orchestrator or planner to avoid duplication.
